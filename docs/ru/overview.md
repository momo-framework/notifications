# Обзор

`momo/notifications` — унифицированный слой исходящих сообщений Momo. Один
объект-нотификация описывает, *что* сообщить получателю; пакет решает, *как* —
веером по email, SMS, push, in-app inbox, Slack и произвольным webhook'ам — не
раскрывая вызывающему коду деталей транспорта.

## Поток

```
$dispatcher->send($user, new OrderShipped())
        │
        ▼  push SendNotificationJob
   очередь Momo ──▶ воркер ──▶ dispatcher->sendNow()
                                   │  для каждого канала из via():
                                   ├─ throttled?  → NotificationThrottled, пропуск
                                   ├─ channel->send() доставил? → NotificationSent
                                   └─ бросил?      → NotificationFailed, проброс (очередь повторит)
```

`send()` ставит в очередь; воркер позже вызывает `sendNow()`, который выполняет
фактическую доставку по каналам. `sendNow()` можно вызвать и напрямую для
синхронной отправки (например, код подтверждения, который должен уйти в рамках
запроса).

## Строительные блоки

| Концепция     | Тип                                   | Ответственность                                  |
|---------------|---------------------------------------|--------------------------------------------------|
| Нотификация   | `NotificationInterface` + `RoutesTo*` | объявляет каналы и строит каждое сообщение        |
| Получатель    | `NotifiableInterface` / `Notifiable`  | идентичность, адрес по каналу, предпочитаемая локаль |
| Канал         | `NotificationChannelInterface`        | превращает нотификацию в вызов транспорта         |
| Транспорт     | `*TransportInterface`                 | механизм доставки (заменяется на реальный)        |
| Диспетчер     | `NotificationDispatcherInterface`     | маршрутизация, throttling, события, async         |
| Throttler     | `ThrottlerInterface`                  | ограничение частоты на получателя                 |
| Inbox         | `NotificationStoreInterface`          | хранилище для in-app канала `broadcast`           |

## Проектные решения

- **Интерфейсы-билдеры на канал, а не толстый базовый класс.** Нотификация
  реализует `RoutesToMail`, `RoutesToSms`, … только для поддерживаемых каналов.
  Канал использует `instanceof` для проверки применимости — полностью типобезопасно,
  без `method_exists`-рефлексии и без `mixed`.
- **`send()` возвращает bool.** Канал сообщает, доставил ли он на самом деле
  (или пропустил из-за отсутствия маршрута/билдера), поэтому диспетчер публикует
  `NotificationSent` только при реальной доставке.
- **Очередь по умолчанию.** Диспетчеризация по умолчанию асинхронна через
  `momo/queue`, держа горячий путь запроса свободным от задержек SMTP/HTTP. См.
  заметку про at-least-once в `SendNotificationJob` о семантике повторов для
  многоканальных нотификаций.
- **Транспорты — интерфейсы с in-memory дефолтами.** Поставляемые
  `Array*Transport` реальны, протестированы и полезны для локальной разработки и
  тестов; в проде связываются SMTP, Twilio, FCM, корутинный HTTP-клиент и т.д.

## Локализация

`NotifiableInterface::preferredLocale()` отдаёт локаль получателя; билдеры
сообщений читают её (например, `MailMessage::$locale`) для локализованного
контента. Полноценный перевод — отдельный i18n-пакет; этот слой лишь
прокидывает локаль.

## Корутино- и память-безопасность

Дефолтные throttler и in-app store держат состояние на время жизни воркера, не
на запрос, и их мутирующие методы не содержат точек приостановки I/O — атомарны
относительно соседних корутин в одном воркере Swoole. Для межворкерной
корректности (общие счётчики throttle, устойчивый inbox) свяжите реализации
`ThrottlerInterface` и `NotificationStoreInterface` на Redis/БД.
